## What is Crowd Funding

Crowd funding is a way to collect small amounts of money from many people to
reach one big goal. Someone creates a campaign and says how much money they
need and by when. Other people send money to that campaign if they like the
idea. If the campaign reaches its goal before the deadline, the owner of the
campaign can take the money out. If the campaign does not reach the goal
before the deadline, the people who sent money should be able to get their
money back.

## Basic Features in the Smart Contract

These are the basic features I want in the first version of the contract:

- Create a campaign
  - A user can create a new campaign by giving a title, a description, a
    funding goal (target amount), and a deadline.
- Contribute to a campaign
  - Any user can send ETH to a campaign to support it, before the deadline.
- Track contributions
  - The contract should keep track of how much each address has contributed
    to a campaign.
- Withdraw funds
  - If a campaign reaches its goal, the campaign owner can withdraw the
    collected funds.
- Refund contributors
  - If a campaign does not reach its goal by the deadline, contributors can
    ask for their money back.
- View campaign details
  - Anyone can read the details of a campaign, such as owner, goal, deadline,
    amount raised, and status (active, successful, failed).
- List all campaigns
  - The contract should let us get a list of all campaigns that were created.

I may add more features later, like letting the owner cancel a campaign or
adding campaign categories, but the list above is enough for a first version.

## How the Backend Will Talk to the Contract

The backend will be written in Golang. It will not hold user funds, it will
only read data from the contract and help the frontend send transactions.
Basic plan:

- Use a Go Ethereum client library (like go-ethereum) to connect to the
  blockchain node (for example, a local Anvil node from Foundry, or a
  testnet).
- Read the contract ABI (generated by Foundry) so the backend knows how to
  call the contract functions.
- Expose simple REST API endpoints, for example:
  - Get list of all campaigns
  - Get details of one campaign
  - Get contributions made by a specific address
- For actions that change state (create campaign, contribute, withdraw,
  refund), the actual transaction will be signed and sent from the frontend
  using the user's own wallet (like MetaMask). The backend will mostly be used
  for reading and showing data, not for signing transactions on behalf of
  users.
- The backend can also listen to contract events (like CampaignCreated or
  ContributionMade) and store them, so the frontend can load data faster
  without asking the blockchain every time.

## Basic Features in the UI (ReactJS)

- Connect wallet
  - Let the user connect their wallet (MetaMask) to the site.
- View all campaigns
  - Show a list of all campaigns with basic info: title, goal, amount raised,
    and deadline.
- View one campaign
  - Show full details of a single campaign, including description and a list
    of contributions.
- Create a campaign
  - A form where the user can enter title, description, goal, and deadline,
    then send the transaction to create a new campaign.
- Contribute to a campaign
  - A form where the user can enter an amount and send ETH to a campaign.
- Withdraw funds (for campaign owner)
  - A button for the campaign owner to withdraw funds once the goal is met.
- Claim refund (for contributors)
  - A button for contributors to claim a refund if the campaign failed.
- Show transaction status
  - Show simple messages like "pending", "success", or "failed" while a
    transaction is being processed.

## V2: Learning Distributed Systems and Microservices

Everything above (contract, backend, frontend) is one monolith today. I want to
start learning how to build a distributed system with a microservice
architecture, but I don't want to rewrite everything at once — I want to add
one small, focused microservice at a time so I actually learn instead of just
generating a big system I don't understand.

The first microservice I'm building: **Comments Service**.

- Right now, comments on a campaign page are fake — just hardcoded seed data in
  the frontend (`SEED_COMMENTS`), not connected to any backend at all.
- Comments are a good first microservice because they don't touch money or the
  smart contract at all — a comment only needs a campaign id, an author, some
  text, and optionally a parent comment id (for replies). It's a clean, low-risk
  bounded context, separate from the main backend's campaign/contract logic.
- It will be its own service, with its own database, talking to the main
  backend over gRPC (not REST) so I can learn gRPC directly.
- The main backend will act as the "front door" for the browser as before
  (frontend keeps calling normal REST endpoints on the main backend), but for
  comments, the main backend will be a gRPC *client* calling the new Comments
  service internally. This is a common real-world pattern (backend-for-frontend
  talking to internal services over gRPC), and it means the browser never needs
  to speak gRPC directly.

Other microservice ideas I want to come back to later, once comments are
working end to end: a notification service, a search/discovery service, an
analytics/reporting service, and pulling the media/asset upload logic out into
its own service (a good candidate for learning client-streaming gRPC).
